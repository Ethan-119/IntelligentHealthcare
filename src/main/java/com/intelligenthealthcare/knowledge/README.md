# knowledge 模块说明

本模块负责医疗知识库基础数据（疾病、医疗能力、医院、科室、医生等）在系统内的实体与仓储接入。

## 数据库职责更正（生效口径）

### 1）PostgreSQL（主业务数据库）

作用：存结构化业务数据、关系数据、导入任务、用户、会话、日志。

主要存放：
- `patient`（用户）
- 导诊会话、消息、槽位
- `import_job` 导入任务、失败日志、审核项
- 各种关系映射表
- AI 审计日志

特点：事务、强一致性、与 DDD 领域模型天然适配。

### 2）MongoDB（向量知识库 + 文档数据库）

作用：专门存 AI 导诊医疗知识库与向量检索数据。

主要存放：
- 疾病
- 医疗能力
- 医院
- 科室
- 医生

约束：knowledge 相关基础数据后续统一按该口径演进，不在业务模块重复定义实体。

### 3）Redis（后续添加）

当前先不引入 Redis；缓存与会话增强后续按独立方案补充。

## 模块定位（请务必遵守）

- `knowledge.domain.model` 下实体是基础数据载体，不承载业务规则与复杂领域行为。
- 导入、审核、临床规则放在对应业务上下文（如 `importjob`），通过 `import` 使用 knowledge 实体。
- 不复制 knowledge 实体到其他模块。
# knowledge 模块说明

本模块提供**医疗基础知识与向量检索**相关的**表映射实体**与 **MyBatis Mapper**，作为全库共享的**基础数据层**。

## 定位（请务必遵守）

- 疾病、医院、科室、医生、能力关系等实体属于**基础数据**，用于落库、查询与**向量字段**维护。
- 这些类**不**作为承载业务规则、领域行为、用例流程的聚合根；**不要**在 `knowledge.domain.model` 中新增业务方法或复杂领域逻辑。
- 与导入、审核、临床决策相关的行为应放在对应业务模块（例如 `importjob` 中解析与策略，通过 `import` 使用本模块实体，而非在本模块内堆逻辑）。

## 包结构（当前）

- `domain.model`：与表结构对应的实体（Lombok 等可保留为数据类写法）。
- `infrastructure.persistence`：各表 `Mapper` 及 MyBatis 接入。

已移除无代码占位的空包：`api`、`application`、`domain.repository`（原仅 `package-info`）。
